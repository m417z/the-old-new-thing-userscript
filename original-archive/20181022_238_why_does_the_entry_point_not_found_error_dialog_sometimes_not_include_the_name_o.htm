Why does the Entry Point Not Found error dialog sometimes not include the name of the missing entry point?

Date:October 22, 2018 / year-entry #238
Tags:code
Orig Link:https://blogs.msdn.microsoft.com/oldnewthing/20181022-00/?p=100025
Comments:    29
Summary:Because the name is too long.

A customer observed that sometimes the Entry Point Not Found error dialog does not include the name of the missing entry point.

Usually, it looks like this:

Contoso.exe - Entry Point Not Found
⊗ The procedure entry point WidgetStart could not be located in the dynamic link library widget.dll.
OK

But sometimes it looks like this:

Contoso.exe - Entry Point Not Found
⊗ The procedure entry point  could not be located in the dynamic link library widget.dll.
OK

Why does the error message sometimes leave out the name of the missing entry point?

If the entry point name is longer than 128 characters, then it is omitted from the error message. The most common reason for having a very long function name is that you exported decorated C++ functions, because C++ function decoration can result in very long names.

    That error dialog was somewhat useful up to Windows 7. Many times, a given error message shares the same root cause across different systems. Starting with Windows 8, the message is misleading at least: it says the executable module that tried to load the procedure, not the module in which the entry point was expected to be found. I believe the routine that builds the error message mistakenly gets the wrong piece of information because heavy internal changes were made to the DLL loader in the Windows 8 timeframe.

    I wonder how many of you remember the __crtCreateSymbolicLinkW issue. I was puzzled a few years ago when users reported in forums that “the entry point XXXX could not be located in MSVCP110.dll”. I expected the library to be MSVCR110.dll. Between Visual C++ 2012 RTM and Visual C++ 2012 Update 1, some wrapper functions were added to the C Runtime. The RTM version of MSVCR110.dll didn’t have these wrappers, but the Update 1 version of the C++ Runtime (MSVCP110.dll) relied on some of them. The error message appears when an Update 1 (or later) Msvcp110.dll library and a RTM Msvcr110.dll are loaded in the same process. This breaking change prompted Microsoft to release an updated Visual C++ 2012 Runtime package through Windows Update.
