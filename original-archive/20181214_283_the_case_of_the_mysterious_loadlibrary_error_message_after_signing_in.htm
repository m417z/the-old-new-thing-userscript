The case of the mysterious LoadLibrary error message after signing in

Tags:other

Comments:    24
Summary:A little screen shot forensics.

A customer reported a Windows bug via Feedback Hub. It said that after signing in, they got this mysterious error message:

Error ×
LoadLibrary failed with error 87: The parameter is incorrect.
OK

The customer included a screen shot but no diagnostic information or process dump files. This is all you get:

📷
Contoso
Camera

📖
LitWare

🚀

📄
random
docum…

📄
another
docum…

Error ×
LoadLibrary failed with error 87: The parameter is incorrect.
OK
📁
📷
Error

7:00 AM
12/14/2018
🗨

You are now in an episode of CSI: Crime Scene Investigation. The investigator has only a tiny piece of physical evidence and must use it to track down the killer.

Can you solve it?

Look closely at the taskbar. The button for the error dialog has an unusual icon.

📷 Error

It's a camera, which happens to match the camera icon on the desktop for Contoso Camera.

This error dialog didn't come from Windows. It came from Contoso Camera. The customer can take up the issue with Contoso.

Now, it's entirely possible that something in Windows changed that caused Contoso Camera to run into problems, but we'll let the Contoso Camera folks take a crack at the problem first, seeing as they understand their code better than we do.

      1. warrens says:
        December 14, 2018 at 1:27 pm

        This wouldn’t work for a variety of reasons…

        1) Stack trace won’t be of much use if you don’t have symbols.
        2) You might be able to get the location of the call to MessageBox(), but not the location of the error itself, and certainly not any variable information that would help with debugging. There might be one MessageBox() call to cover many possible error conditions.
        3) MessageBox() might be called on a separate thread so that it doesn’t block your UI thread!
        4) And that’s just the Win32 MessageBox() API. There are different modal message box implementation in each of Windows Forms and WPF, which means now you’re dealing with generating stack traces in .NET, which has a whole other set of issues.
        5) The UWP MessageDialog implementation is async, which makes getting an accurate stack trace even more difficult.

        Oh, and also, you can just right-click on the message box’s taskbar item, then right-click the name of the application, then hit Properties. Now at least you’ll know the thing you really want to know — who’s responsible?
